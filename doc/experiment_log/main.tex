\begin{abstract}
DFX4ML-ARCH presents an FPGA architecture designed to enable partial reconfiguration for machine learning workloads. The project incorporates a self-reconfiguration mechanism, allowing the FPGA fabric to partially reprogram itself. It targets AMD FPGAs equipped with the ICAP3 interface and leverages the PYNQ framework for system integration.
\end{abstract}

\section{Introduction}

The goal of this experiment session is to gather empirical data on the key parameters that govern design trade-offs in an FPGA partial reconfiguration system for machine learning workloads. The results will inform architectural decisions going forward.

\subsection{Parameters Under Consideration}

\begin{description}
  \item[Data Batch Size]
    Varies with static region size and model I/O type.
    A larger batch size is desirable, but it competes directly with the desire for a large static region.

  \item[FPGA Static Region Size]
    Should be as small as possible to leave more area for the dynamic region.

  \item[FPGA Dynamic Region Size]
    Should be as large as possible to accommodate larger ML models.
    However, a larger dynamic region increases reconfiguration time.

  \item[Reconfiguration Time]
    Should be as small as possible, which pushes toward a smaller dynamic region --- directly opposing the goal above.

  \item[Model Size]
    Larger ML models are preferred, but they require a larger dynamic region, which increases reconfiguration time.

\end{description}

\subsection{Goal}

Quantify the actual trends between these parameters so that architectural decisions can be grounded in evidence rather than assumption. Three experiment sessions are proposed:

\begin{enumerate}

  \item \textbf{Region Size vs. Reconfigure Time} ---
    The reconfigure time depends on The size of dynamic region (\textbf{not size of model in dynamic region}). The experiment aims to explore the trend of time consumming in varies size of dynamic region

  \item \textbf{DFX Streamer packing exploration} ---
    We proposed the optimized way to expand the size of dynamic region as much as possible while keep the BRAM/URAM in static region for DFX Streamer as much as possible as well.

  \item \textbf{Latency Hiding Technique (IN PROGRESS)} ---
    We proposed the technique to hiding the latency by using the double dynamic region method to keep the system executing all the time.


\end{enumerate}

With data from all sessions, the aim is to identify the operating point that maximises performance, or at minimum to establish a principled basis for deciding the next phase of the design.



\section{Architecture Upgrade}

\subsection{DFX Streamer}


\subsubsection{Problems}

We encounter 2 problems on Vivado and KV260:

\begin{enumerate}
  \item \textbf{Vivado cannot handle a single large BRAM/URAM array.}
    Vivado synthesis rejects the \texttt{mainMem} variable in \texttt{dfx\_streamer.v} (line~82) with the following error:

    \begin{quote}
      \ttfamily\small
      [Synth 8-4556] size of variable `mainMem' is too large to handle; \\
      the size of the variable is 20{,}480{,}000, the limit is 1{,}000{,}000.
    \end{quote}

    The variable \texttt{mainMem} is declared as a flat Verilog register array whose total bit-width (20\,480\,000) exceeds Vivado's synthesis limit of 1\,000\,000 bits.
    Vivado cannot map such a monolithic array to physical BRAM/URAM primitives directly.
    The fix is to split \texttt{mainMem} into multiple smaller arrays, each within the 1\,000\,000-bit limit, so that Vivado can infer and place individual BRAM/URAM tiles.

  \item \textbf{Boarder Latency Problem}
    The old version of DFX Streamer (Magic Streamer) requires one cycle for save the data to BRAM/URAM. The timming does not met. My Assumption is the placement restriction from different Pblock. Fig. \ref{fig:bram_problem} illustrates the SRC signal (green block) which is restricted in dynamic region while the target bram is spread across another region. the Vivado cannot do the fan-out optimization accross the region especially in the dynamic region. Morever, Vivado cannot retimming the path as well.

    \begin{figure}[H]
      \centering
      \includegraphics[width=0.55\linewidth]{bram_problem.png}
      \caption{Vivado synthesis error caused by oversized \texttt{mainMem} array in \texttt{dfx\_streamer.v}.}
      \label{fig:bram_problem}
    \end{figure}
 
\end{enumerate}
 
\subsubsection{DFX Stream Next Gen}

We address the DFX Streamer's problem by fully re-architecture the memory Management System.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.65\linewidth]{dfx_streamer2.png}
  \caption{DFX Streamer next-generation architecture.}
  \label{fig:dfx_streamer2}
\end{figure}


First, to address the Vivado error message, we divide the DFX Streamer's memory bank into multiple bank. Each bank have IO with equal to the IO of the ML model's front wrap. However, multiple of 64 bit aligment wrap width is recommended because URAM's bitwidth equal to 64 bit. 

Moreover, The DFX Streamer will try to pack 4096 entries foreach bank because we aims to maximise the resource usage for each URAM. If we did not pack the entries, the vivado cannot decide to join two Memory bank into into single URAM/BRAM eventhough the block is not accessed at the same time.

Second, to address the latency problem, we do pipelining on the URAM/BRAM data transfer (both load and store). On the store side, we add addition register with equal to wrap size to buffer the data come from the dynamic region. On the other side, we add the addition register with equal to $wrap size x amt of Mem bank$, the pipeline support the back pressure in case the ML model is not ready to receive the data from memory bank.


\subsection{DFX Manager Next Gen}

In current experiment session, we used the ML model from HLS4ML's Vitis Unified Backend; therefore we need the AXI-LITE for control the MODEL. We have added two register into BANK 0 of the \texttt{s\_axi\_write} module:

\begin{itemize}
  \item \textbf{Address DFX\_MAN\_ADDR + \texttt{0x240}} --- \reg{prCtrlAddr}: stores the base address of the PR control IP (AXI-Lite slave of the HLS4ML model).
  \item \textbf{Address DFX\_MAN\_ADDR + \texttt{0x280}} --- \reg{batchSize}: stores the batch size used for a single inference invocation.
\end{itemize}

Both registers are 32-bit wide (\texttt{GLOB\_ADDR\_WIDTH} / \texttt{GLOB\_DATA\_WIDTH}) and are decoded from address bits~[13:6] in the BANK~0 region (\texttt{write\_addr[15:14] == 2'b00}).

To accurately collect the execution time for each ML, we added the 32 bits register into Bank 1 (\textbf{\texttt{0x4024 + 0x64 * RM\_ID}}) to collect the amount of cycles in the system.

\section{Temporal HLS4ML's Vitis Modified Backend Modification}

To explore multiple design that fit into DFX4ML and KV260, we decided to modify the HLS4ML's Vitis Unified backend to enable the it's interface connected to DFX4ML system. However, we did not want to build full new back-end right now. We want the minimal modification as follow:

\begin{enumerate}
  \item the mod backend's intermediate the layer's warp as baremetal bits (not convert back to float32 or other type)
  \item we use software prediction result from Old Vitis Unified Backend for result checking.
  \item we use Fifo-Depth optimization from Old Vitis Unfied Backend and manually copy it into the new chopped model
\end{enumerate}

\begin{sloppypar}
the modified backend is at \url{https://github.com/Tanawin1701d/hls4ml/tree/VitisUnifiedPartialExp}
\end{sloppypar}


\subsection{Interface Configuration}

The modified backend is invoked via the following call:

\begin{lstlisting}[style=python]
hls_model = hls4ml.converters.convert_from_keras_model(
    keras_model,
    hls_config=cfg,
    output_dir=output_dir,
    input_flat=input_flat,
    output_flat=output_flat,
    package_as_xo=full_run,
    project_name=_PROJECT_NAMES[key],
    **_HLS_PARAMS,
)
\end{lstlisting}
 
Three parameters are specific to the modified backend:

\begin{description}
  \item[\texttt{input\_flat}]
    A boolean flag that controls whether the model's input warp is flattened into a 1-D stream of baremetal bits before being passed to the HLS kernel.
    When \texttt{True}, it is flattened, else \texttt{False} the system convert from float32 to the target tensor.

  \item[\texttt{output\_flat}]
    The output-side counterpart of \texttt{input\_flat}.

  \item[\texttt{package\_as\_xo}]
    When \texttt{True} (bound to \texttt{full\_run}), the backend runs the full new Vitis HLS flow and packages the resulting kernel as an \texttt{.xo} object file ready for linking.
    When \texttt{False}, it system will follow Vitis HLS ip for vivado flow.
\end{description}

\begin{sloppypar}
The in detail implementation of the modified interface is at \url{https://github.com/Tanawin1701d/hls4ml/blob/VitisUnifiedPartialExp/hls4ml/templates/vitis_unified/myproject_axi_stream.cpp}
\end{sloppypar}

\textbf{Key Takeaway}: As I experiment last year, the ZCU102's reconfigure rate is limited to ~399 MB/sec; therefore, the bottleneck is caused from the ICAP3 port that can be accepted only 32 bits per cycle (4 bytes). So at 100 MHz, the system's rate therefore saturate at 400 MHz (4 bytes * 100 MHz)

In addition, the bit file size and reconfigure time did not depend on the amount of used logic cell in the FPGA, it depends on the region size in the fabric.

\subsection{DFX Streamer Packing Exploration}


\subsubsection{Dynamic Region Layout}

To allocate the fabric's resource for both dynamic region and static region.

\begin{enumerate}
  \item Static Region: we need it small as much as possible because it is not relevant to the ML workloads. However, we need the on-chip memory as much as possible because we require it for DFX Streamer to hold the intermediate data.
  \item Dynamic Region: we need it large as much as possible because it is mainly used for ML workloads. However, The large region may invade the static region's URAM/BRAM memory and increase the reconfigure time.
\end{enumerate}


As a result, we design the region arrangement shown in Fig.~\ref{fig:packing_layout}.
We compactise the static region into the right of the fabric which has plenty of URAM/BRAM.
