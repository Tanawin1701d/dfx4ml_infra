\begin{abstract}
DFX4ML-ARCH presents an FPGA architecture designed to enable partial reconfiguration for machine learning workloads. The project incorporates a self-reconfiguration mechanism, allowing the FPGA fabric to partially reprogram itself. It targets AMD FPGAs equipped with the ICAP3 interface and leverages the PYNQ framework for system integration.
\end{abstract}

\section{Project Overview}

DFX4ML is the FPGA architecture aimming to execute FPGA's machine learning workload by exploiting partial reconfiguration to fit FPGA's resource to large machine learning model.
The model will be divided into multiple peice and then sequentially set up on Reconfigurable region to execute the machine learning workload.

This project is composed of sections:
    \begin{enumerate}
    \item \textbf{Hardware Architecture}:  it describe architecture idea, hardware implementation detail, reconfigurable parameter.
    \item \textbf{Software Driver}: This driver mainly controls our hardware IP and Xilinx IP in the programable logic. In this version, we provide only the pynq driver.
    \item \textbf{ML framework integration}: This section provides our formal approach to enable the DFX4ML-ARCH to exist Hardware Machine Learning Framework such as HLS4ML.
    \end{enumerate}

\section{Hardware Architecture}

\begin{figure}[ht]
\centering
\includegraphics[width=0.8\textwidth]{images/overall_system.png}
\caption{\label{fig:overviewArch} DFX4ML-ARCH overall architecture}
\end{figure}

Fig.\ref{fig:overviewArch} illustrates the overall hardware architecture of DFX4ML-ARCH. The hardware architecture is composed of 4 major part:

\begin{enumerate}
\item \textbf{Main Processor}. This project focuses on Xilinx SOC with PYNQ framework. The PYNQ framework supports users to manage memory allocation and initialization. Moreover, the processor is responsible for controlling Programable logic and communicating to the PMU processor.
\item \textbf{PMU and CSU}. Plateform Management Unit (PMU) and Control Security Unit (CSU) is the Xilinx SOC's management system. The PMU is the dedicate processor used to manage system's routine task and secure the system at hardware level. In this project, we have to command the PMU to orchestrates the Programmable logic permission.
\item \textbf{Programmable Logic}. To support partial reconfiguration, the DFX4ML-ARCH is divided into 2 region:
  \begin{enumerate}
    \item \textbf{Static Region}. The region that is always active and contains the control logic and temporal memory storage. We propose four main component:
      \begin{enumerate}
        \item \textbf{DFX Manager}: it manages all PL components: (1) command DFX Controller, (2) control DFX Streamer, (3) control DMA Controller, (4) profile thte system, (5) interrupt the processor, and (6) receive commands and meta-data from processor.
        \item \textbf{DFX Controller}: This Xilinx IP recieve command from DFX Manager. When It is triggered, the system will load the partial bit stream file from main memory (DDR ram) and reprogram the partial region through ICAP 3 port.
        \item \textbf{DFX Streamer(s)}: it is the storage based on BRAM (on-ship memory) used to hold the data temporarily while the patial region is reconfiguring.
        \item \textbf{DMA Controller}: this Xilinx IP is used to transfer Data from DDR ram directly to partial region.
      \end{enumerate}
    \item \textbf{Reconfigurable Region (RP)}. This region is dedicated to machine learning workload. The input\/output only comes from the AXI-Stream.
  \end{enumerate}

\end{enumerate}



 
\subsection{DFX Manager}

DFX Manager is the IP used to orchestrates entire Programmable Logic function. Fig. \ref{fig:dfx_mng_arch} illustrates the overall components in the manager.

\begin{figure}[ht]
\centering
\includegraphics[width=0.8\textwidth]{images/dfx_mng_arch.png}
\caption{\label{fig:dfx_mng_arch} DFX Manager}
\end{figure}


The DFX Manager is composed with 5 major components:

\begin{enumerate}
  \item PS connnector: It communicates with the main processor (PS) through AXI-LITE slave protocol. The code is at \textcolor{darkgreen}{\texttt{hw/ip\_src/dfx\_mng/s\_axi\_\{read,write\}.v}}
  \item DFX Streamer Connector: The connector is directly passed the signal from the DFX Manager Core through direct control signal to trigger load/store operation, or even receive the data holding status from the streamers.
  \item DMA Connector: The DFX Manager Core requires multiple step to set-up the DMA controller properly; therefore, the DMA connector is responsible to manage the sequence of AXI-LITE packet sending DMA IP. The code is at\newline\textcolor{darkgreen}{\texttt{hw/ip\_src/dfx\_mng/m\_axi\_\{read,write\}.v}}
  \item DFX CTRL Connector: The connector is directly passed the signal from the DFX Manager Core through direct control signal to trigger the DFX CTRL IP to load to decouple and reconfigure the partial region zone. This connector also get the data back from the target IP to indicate that the reconfiguration is finished.
  \item DFX Manager Core: it contains the internal register memory that used to control the flow of the system and provide the data for the slave IP. Moreover, Flow CTRL is responsible to orchestrates the slave IP to work in the correct order.
\end{enumerate}

\subsubsection{DFX Manager Core}

\paragraph{\textbf{Register Architecture (internal register bank)}}

\begin{figure}[ht]
\centering
\includegraphics[width=1.0\textwidth]{images/dfx_mng_addr_tran.png}
\caption{\label{fig:dfx_mng_tran} DFX Manager Core's internal register address translation}
\end{figure}

We propose the register architecture that resides in the internal memory bank. Users can access these via memory map IO. The tcl building script set the default of the address at $0xA004\_0000$ with 64 KB address range.

With 64 KB range, The address alignment is categorized into 3 main section:

\begin{enumerate}
  \item \textbf{bankId} (bit 14-15): the system supports up to 4 banks. Each bank is a group of row that belongs to the same group of purpose.
  \item \textbf{rowId} (bit 13-6): Each bank supports up to 256 rows. Row is the group of registers, however, the different row in the same bank have not to be the same schematic.
  \item \textbf{column} (bit 5-2): Each row supports up to 16 register. Each register in the row has not to be the same register size. However, the register's size must not exceed 4 bytes (32 bit)
\end{enumerate}

Fig. \ref{fig:dfx_mng_reg} illustrates the DFX Manager's intenal memory. the system is composed of two memory bank: Bank 0 is dedicated for the IP control, debugging system, overall running configuration, interrupt setting, flow control memory; Bank 1 is composed serveral same register pattern row used to store data and meta-data for reconfigurable module initialization, and slave static IP command.

\paragraph{\textbf{Flow Control (state machine)}}

Flow Control orchestrates the slave IP step such as DFX Controller, DMA IP, DFX Streamer (formerly Magic Streamer) to execute intilization and operation command sequentially at cycle-accurate level. The code is at $hw/ip\_src/dfx\_mng/dfx\_mng\_core.v$.

Fig. \ref{fig:dfx_mng_state} illustrates the high-level state machine inside the DFX Mng Core, and we can simply categorized the system command into 3 main parts:

\begin{enumerate}
  \item SHUTDOWN state: The DFX Manager will not issue any command and not follow any command from outside source except the command from processor. All internal registers can be read or write followed by the rule at previous section (Access Permission Legend). Except the \reg{CTRL\_REG}, if there are register accesses when the system is not in SHUTDOWN state, the system will send back the unpredictable value and our driver will discard it. We apply this restriction because we aim to optmize the hardware resouce usage.
  \item DFX CTRL states: these states is depicted in DFX CTRL hanshake section (green block) and dedicates to send the commands to trigger DFX CTRL IP. The commands aims to reconfigure the partial region, and track/synchronize the status of the reconfiguring.
  \item MGS and DMA states: these states is depicted in MGS \& DMA initialization (brown block) and deicates to send the commands to initialize and trigger the DMA IP and DFX Streamer (formerly magic streamer (MGS)). The commands aims to prepare/clear the internal meta-data memory of DFX Streamer, set-up the DMA IP meta-data that get from internal register bank 1, and wait for finish from reconfigurable module.
\end{enumerate}

\subsection{DFX Streamer(s)}

DFX Streamer(s) is the temporal storage reside in the Programable logic's static region and it was designed to utilize the BRAM resource for the storage and minor flip flop for holding its tranfer meta-data. the code is at \textcolor{darkgreen}{\texttt{hw/ip\_src/dfx\_streamer/dfx\_streamer.v}}

\begin{figure}[H]
\centering
\includegraphics[width=0.65\textwidth]{images/dfx_streamer.png}
\caption{\label{fig:dfx_streamer} DFX Streamer Architecture}
\end{figure}

Fig. \ref{fig:dfx_streamer} illustrates the overal architecture of the DFX Streamer, the \textbf{meta-data blocks} (blue boxes) stay inside the static region using flip flop to track amount of load/ store and RM status (INTERRUPT HOLDER). Meanwhile the \textbf{MAIN MEMORY} (purple block) also resides in the static region, however it uses BRAM memory to store the temporal data.

In contrast, the Stream Transceiver and Stream Receiver' connection port are open to partial region with \textbf{AXI-STREAM} interface. In current version, AXI\_STREAM's \textbf{tkeep}, and \textbf{tlast} signal are \textbf{mandatory}. If they is not set-up, the system will get stuck.

\subsection{DFX\_CTRL}

DFX\_CTRL is the Xilinx's DFX IP used to retrive the bin stream file from DDR Memory and reconfigure the partial reconfigurable region by using the ICAP3 port. To achieve this process, we need to configure the IP's internal memory to match our feature.

\begin{figure}[H]
\centering
\includegraphics[width=0.65\textwidth]{images/dfx_ctrl_register.png}
\caption{\label{fig:dfx_streamer} DFX Controller Architecture}
\end{figure}

Fig. \ref{fig:dfx_streamer} illustrates the internal memory of the DFX Ctrl IP. It is composed of 4 memory bank: 

\begin{enumerate}
  \item Bank 0 is general register enabling the software the control IP and check IP status.
  \item Bank 1 is the RM Mapping, each slot in this bank is the pointer to a slot in Bank 2.
  \item Bank 2 is the RM information. The control field is the setting of the start-up/shutdown down procedure, and our driver set this criteria as follow:
      \begin{enumerate}
        \item Reset Duration: 1 cycle
        \item Reset Type: active low
        \item Startup Required: No
        \item Shutdown Required: No
      \end{enumerate}
      The BS Index field contains the index of the target slot in bank 3
  \item Bank 3 contains the bitstream infomation: 
      \begin{enumerate}
        \item Address: the address of bitstream file in DDR Memory
        \item Size: the size of the bitstream file in bytes
        \item ID: bit stream Identifier
      \end{enumerate}
\end{enumerate}

 \subsection{ML Framwork Integration}

 DFX4ML-ARCH provides the official solution for enabling Hardware Machine Learning framework to integrate their own sub-kernel into each Reconfigurable Module (RM).

To integrate the module, users must complete the following steps:
\begin{enumerate}
    \item Provide the path to the user IP repository via \texttt{user\_repo\_path}. The folder at that path must contain the Vivado-exported IP folder, which in turn must include \texttt{src} and \texttt{xgui} sub-folders.
    \item Provide the path to the TCL file that instructs the build script how to build the Reconfigurable Module via \texttt{user\_build\_tcl} (Listing~\ref{lst:python_build}).
    \item Set \texttt{test\_mode} to \texttt{0} (Listing~\ref{lst:python_build}) to enable the actual kernel instead of the loopback test logic.
    \item Provide the IP name list in $ip\_map\_list$ (Listing~\ref{lst:python_build}). The index is the Reconfigurable Module and the value is the IP name.
\end{enumerate}

\subsection{Board Specific Script}

In hardware formation of this current version, Only Kv260 board can be fully synthesis and implementation. If the $board$ parameter is set to $custom$, the hardware formation will stop at DFX block design with DFX Unified and Partial connection, the PS and Connection will be missing.

To define new board, users have to build/modify two file
  \begin{enumerate}
    \item board\_file\: users have to create the file at \newline\textcolor{darkgreen}{\texttt{hw/build\_script/<your board name>/board\_build.tcl}} \newline The purpose of this file is to give a processor block, inter-connect block, interrupt system, and other relevent IP in the Vivado block design. Users can use \newline \textcolor{darkgreen}{\texttt{hw/build\_script/kv260/board\_build.tcl}} or use GUI in bord $custom$ as an reference.
    \item constraint\_file\: users have to create the file at \newline\textcolor{darkgreen}{\texttt{hw/build\_script/<your board name>/constraint.xdc}} The purpose of this file is to give the boundary of partial reconfigurable region and its resources. Users can use \newline\textcolor{darkgreen}{\texttt{hw/build\_script/kv260/constraint.xdc}}
    \item build\_script\: users have to modify the file at \newline\textcolor{darkgreen}{\texttt{hw/build\_script/build.tcl}} to invoke the block design creation procedure. Users may use kv260 option as a reference.
  \end{enumerate}

DFX4ML-ARCH's author would be appreciated if users do pull request for specific board file.
