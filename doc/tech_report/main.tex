\begin{abstract}
DFX4ML-ARCH presents an FPGA architecture designed to enable partial reconfiguration for machine learning workloads. The project incorporates a self-reconfiguration mechanism, allowing the FPGA fabric to partially reprogram itself. It targets AMD FPGAs equipped with the ICAP3 interface and leverages the PYNQ framework for system integration.
\end{abstract}

\section{Project Overview}

DFX4ML is the FPGA architecture aimming to execute FPGA's machine learning workload by exploiting partial reconfiguration to fit FPGA's resource to large machine learning model.
The model will be divided into multiple peice and then sequentially set up on Reconfigurable region to execute the machine learning workload.

This project is composed of sections:
    \begin{enumerate}
    \item \textbf{Hardware Architecture}:  it describe architecture idea, hardware implementation detail, reconfigurable parameter.
    \item \textbf{Software Driver}: This driver mainly controls our hardware IP and Xilinx IP in the programable logic. In this version, we provide only the pynq driver.
    \item \textbf{ML framework integration}: This section provides our formal approach to enable the DFX4ML-ARCH to exist Hardware Machine Learning Framework such as HLS4ML.
    \end{enumerate}

\section{Hardware Architecture}

\begin{figure}[ht]
\centering
\includegraphics[width=0.8\textwidth]{images/overall_system.png}
\caption{\label{fig:overviewArch} DFX4ML-ARCH overall architecture}
\end{figure}

Fig.\ref{fig:overviewArch} illustrates the overall hardware architecture of DFX4ML-ARCH. The hardware architecture is composed of 4 major part:

\begin{enumerate}
\item \textbf{Main Processor}. This project focuses on Xilinx SOC with PYNQ framework. The PYNQ framework supports users to manage memory allocation and initialization. Moreover, the processor is responsible for controlling Programable logic and communicating to the PMU processor.
\item \textbf{PMU and CSU}. Plateform Management Unit (PMU) and Control Security Unit (CSU) is the Xilinx SOC's management system. The PMU is the dedicate processor used to manage system's routine task and secure the system at hardware level. In this project, we have to command the PMU to orchestrates the Programmable logic permission.
\item \textbf{Programmable Logic}. To support partial reconfiguration, the DFX4ML-ARCH is divided into 2 region:
  \begin{enumerate}
    \item \textbf{Static Region}. The region that is always active and contains the control logic and temporal memory storage. We propose four main component:
      \begin{enumerate}
        \item \textbf{DFX Manager}: it manages all PL components: (1) command DFX Controller, (2) control DFX Streamer, (3) control DMA Controller, (4) profile thte system, (5) interrupt the processor, and (6) receive commands and meta-data from processor.
        \item \textbf{DFX Controller}: This Xilinx IP recieve command from DFX Manager. When It is triggered, the system will load the partial bit stream file from main memory (DDR ram) and reprogram the partial region through ICAP 3 port.
        \item \textbf{DFX Streamer(s)}: it is the storage based on BRAM (on-ship memory) used to hold the data temporarily while the patial region is reconfiguring.
        \item \textbf{DMA Controller}: this Xilinx IP is used to transfer Data from DDR ram directly to partial region.
      \end{enumerate}
    \item \textbf{Reconfigurable Region (RP)}. This region is dedicated to machine learning workload. The input\/output only comes from the AXI-Stream.
  \end{enumerate}

\end{enumerate}



 
\subsection{DFX Manager}

DFX Manager is the IP used to orchestrates entire Programmable Logic function. Fig. \ref{fig:dfx_mng_arch} illustrates the overall components in the manager.

\begin{figure}[ht]
\centering
\includegraphics[width=0.8\textwidth]{images/dfx_mng_arch.png}
\caption{\label{fig:dfx_mng_arch} DFX Manager}
\end{figure}


The DFX Manager is composed with 5 major components:

\begin{enumerate}
  \item PS connnector: It communicates with the main processor (PS) through AXI-LITE slave protocol. The code is at \textcolor{darkgreen}{\texttt{hw/ip\_src/dfx\_mng/s\_axi\_\{read,write\}.v}}
  \item DFX Streamer Connector: The connector is directly passed the signal from the DFX Manager Core through direct control signal to trigger load/store operation, or even receive the data holding status from the streamers.
  \item DMA Connector: The DFX Manager Core requires multiple step to set-up the DMA controller properly; therefore, the DMA connector is responsible to manage the sequence of AXI-LITE packet sending DMA IP. The code is at\newline\textcolor{darkgreen}{\texttt{hw/ip\_src/dfx\_mng/m\_axi\_\{read,write\}.v}}
  \item DFX CTRL Connector: The connector is directly passed the signal from the DFX Manager Core through direct control signal to trigger the DFX CTRL IP to load to decouple and reconfigure the partial region zone. This connector also get the data back from the target IP to indicate that the reconfiguration is finished.
  \item DFX Manager Core: it contains the internal register memory that used to control the flow of the system and provide the data for the slave IP. Moreover, Flow CTRL is responsible to orchestrates the slave IP to work in the correct order.
\end{enumerate}

\paragraph{\textbf{Register Architecture (internal register bank)}}

\begin{figure}[ht]
\centering
\includegraphics[width=1.0\textwidth]{images/dfx_mng_addr_tran.png}
\caption{\label{fig:dfx_mng_tran} DFX Manager Core's internal register address translation}
\end{figure}

We propose the register architecture that resides in the internal memory bank. Users can access these via memory map IO. The tcl building script set the default of the address at $0xA004\_0000$ with 64 KB address range.

With 64 KB range, The address alignment is categorized into 3 main section:

\begin{enumerate}
  \item \textbf{bankId} (bit 14-15): the system supports up to 4 banks. Each bank is a group of row that belongs to the same group of purpose.
  \item \textbf{rowId} (bit 13-6): Each bank supports up to 256 rows. Row is the group of registers, however, the different row in the same bank have not to be the same schematic.
  \item \textbf{column} (bit 5-2): Each row supports up to 16 register. Each register in the row has not to be the same register size. However, the register's size must not exceed 4 bytes (32 bit)
\end{enumerate}

Fig. \ref{fig:dfx_mng_reg} illustrates the DFX Manager's intenal memory. the system is composed of two memory bank: Bank 0 is dedicated for the IP control, debugging system, overall running configuration, interrupt setting, flow control memory; Bank 1 is composed serveral same register pattern row used to store data and meta-data for reconfigurable module initialization, and slave static IP command.

\subrow{DMA Transfer — Source}

\addr{+0x00} & \reg{SRC\_ADDRESS} & 32 bit & \badrw &
  Source memory address. Address in system memory from which data will be pushed to the DMA engine. \\

\addr{+0x04} & \reg{SRC\_SIZE} & 26 bit & \badrw &
  Source transfer size in bytes. Specifies the number of bytes to push from source memory to DMA. \\

% ── DMA Destination ───────────────────────────────────────────────────────────
\subrow{DMA Transfer — Destination}

\addr{+0x08} & \reg{DES\_ADDRESS} & 32 bit & \badrw &
  Destination memory address. Address in system memory where the DMA engine writes the output data. \\

\addr{+0x0C} & \reg{DES\_SIZE} & 26 bit & \badrw &
  Destination transfer size in bytes. Specifies the number of bytes the DMA engine writes to destination memory. \\

% ── Status & Profiling ────────────────────────────────────────────────────────
\subrow{Slot Status \& Profiling}

\addr{+0x10} & \reg{SLOT\_STATUS} & 32 bit & \badrw &
Bit~0: load state (0~=~unload, 1~=~loaded). Bit~1: profiler overflow flag
(0~=~overflow, 1~=~no overflow). \\

\addr{+0x14} & \reg{PROFILE\_COUNTER} & 32 bit & \badrw &
  Profiling cycle counter. Counts partial region reconfigure's usage cycles for this RM.\\

% ── Stream Masks ──────────────────────────────────────────────────────────────
\subrow{Stream Masks}

\addr{+0x18} & \reg{LOAD\_MASK} & 8 bit & \badrw &
  Each bit value at each index $i$ in this register refers to the demand of the data from streamer number $i$. However, the first bit (LSB bit) is reserved for DMA. 1 is require; 0 is otherwise\\

\addr{+0x1C} & \reg{STORE\_MASK} & 8 bit & \badrw &
  Each bit value at each index $i$ in this register refers to the request to store data to streamer number $i$. However, the first bit (LSB bit) is reserved for DMA. 1 is require; 0 is otherwise \\


\addr{+0x20} & \reg{STORE\_INTR\_MASK} & 8 bit & \badro &
  It buffers the finish status of the DFX Streamer and DMA IP in the same pattern with \reg{STORE\_MASK}. This register is used to indicate wheather this RM is finished or not\\

\end{longtable}
